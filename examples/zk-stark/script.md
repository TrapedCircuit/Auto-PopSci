### S03 PrereqPolyScene (1:05 - 2:05)

在深入 STARKs 之前，我们需要先认识两个关键工具。第一个是多项式。

[ANIM: 小节标题「预备知识：多项式」]

你一定见过这样的函数：f(x) 等于 2x 的平方加 3x 加 1。这是一个二次多项式。

[ANIM: 函数公式出现 + 曲线绘制]

多项式有一个非常强大的性质。如果两个不同的 d 次多项式，它们最多只能在 d 个点上相交。

[ANIM: 两条曲线，交点高亮]

来看个例子。f(x) 等于 2x 平方加 3x 加 1，g(x) 等于 x 平方加 5x 加 1。它们都是二次的，所以最多只有 2 个交点。

[ANIM: 两条具体曲线绘制，交点标注]

换句话说，如果我在一个很大的范围里随机选一个点去检查，两个不同多项式恰好在这个点上相等的概率极低。

[ANIM: 随机采样点动画，概率文字出现]

这意味着什么？多项式就像指纹——几乎不可能伪造。

[ANIM: 指纹图标 + 多项式曲线叠加]

### S04 PrereqFieldScene (2:05 - 2:55)

第二个关键工具是有限域。

[ANIM: 小节标题「预备知识：有限域」]

你一定知道「取模」运算。比如在模 7 的世界里，3 加 5 等于 1，因为 8 mod 7 等于 1。

[ANIM: 数字 3+5=8, 然后 8 mod 7 = 1 的动画]

在这个世界里，数字只有 0 到 6，做加减乘除全部绕着圈来。这就是一个有限域。

[ANIM: 圆环上 0-6 的数字，箭头演示加法跳跃]

为什么要用有限域？因为计算机处理实数会有精度误差，但模运算是精确的。每一步计算都可以被完美复现。

[ANIM: 浮点误差 vs 精确模运算的对比]

当我们把多项式放在有限域上，就得到了 STARKs 的数学语言。接下来，你会看到它的威力。

[ANIM: 多项式 + 有限域 → STARKs 的组合图标]

## Part 2: 承 -- 核心思想 (2:55 - 5:30)

### S05 CoreIdeaScene (2:55 - 3:50)

现在来看 STARKs 最核心的思想。

[ANIM: 小节标题]

一句话概括：证明一个计算是正确的，等价于证明你知道一个满足特定约束的多项式。

[ANIM: 文字高亮出现]

具体来说，任何计算过程都可以被记录为一张「执行轨迹」表。每一行是一步的状态。

[ANIM: 表格逐行出现]

我们把这张表编码成一个多项式。如果这个多项式满足所有约束条件，就说明原始计算是正确的。

[ANIM: 表格变形成多项式曲线]

而验证者只需要在几个随机点上检查这个多项式，就能以极高的概率确认它是否正确。

[ANIM: 随机点采样 + 概率公式]

### S06 ArithmetizationScene (3:50 - 4:50)

让我们用一个具体的例子来理解。假设要证明一个类似斐波那契的计算。

[ANIM: 小节标题「算术化」]

规则是：a₀ 等于 1，a₁ 等于 1，之后每个数等于前两个数之和。

[ANIM: 数列 1, 1, 2, 3, 5, 8, 13... 逐个出现]

我们要证明自己正确地算到了第 1000 步。

[ANIM: 数列延伸，最后一项高亮]

先来看具体数字。第一个约束：起始值必须是 1。

[ANIM: a₀=1 高亮]

第二个约束：每一步都满足递推关系。比如 a₂ 等于 a₁ 加 a₀，也就是 2 等于 1 加 1。a₃ 等于 a₂ 加 a₁，3 等于 2 加 1。

[ANIM: 具体数字的递推关系逐行展示]

现在把这些值编码到一个多项式 P 上。P 经过点 (0,1)，(1,1)，(2,2)，(3,3)，以此类推。

[ANIM: 散点图 + 多项式插值曲线]

约束条件变成了多项式等式：P 在 x 乘 ω² 处的值，等于 P 在 x 乘 ω 处加上 P 在 x 处的值。

[ANIM: 多项式约束公式出现]

如果 P 在整个特殊域上都满足这个等式——那原始计算就是正确的。

[ANIM: 域上所有点变绿色 ✓]

### S07 PolynomialTestScene (4:50 - 5:30)

但验证者不会逐点检查——那就跟重新计算一样慢了。

[ANIM: 逐点检查 → 叉号]

相反，验证者随机挑几个点抽查。

[ANIM: 随机选点，闪烁]

还记得多项式的指纹性质吗？如果证明者作弊——他提交的不是正确的多项式——那么在随机点上暴露的概率极高。

[ANIM: Schwartz-Zippel 概率公式]

每多检查一个点，作弊成功的概率就指数级下降。这和零知识证明中反复挑战的逻辑一模一样。

## Part 3: 转 -- FRI 协议 (5:30 - 8:00)

### S08 FRIIntroScene (5:30 - 6:20)

但还有一个关键问题：验证者怎么确认证明者提交的东西真的是一个低次多项式，而不是随便编的数据？

[ANIM: 问号 + 低次多项式 vs 随机数据]

这就是 FRI 协议要解决的问题。FRI，全称是 Fast Reed-Solomon Interactive Oracle Proof of Proximity。

[ANIM: FRI 全称展开]

FRI 的核心思想，用一句话说：把多项式反复对折，每折一次度数减半，直到折成一个常数。

[ANIM: 纸条对折动画]

就像你把一张写满数字的纸条反复对折。每折一次，长度减半。折了 log n 次之后，纸条只剩一个点——这一个点你可以直接验证。

[ANIM: 纸条反复折叠，越来越小]

### S09 FRIStepsScene (6:20 - 7:20)

让我们看看 FRI 的具体步骤。

[ANIM: 小节标题「FRI 折叠」]

假设证明者有一个 d 次多项式 f(x)，他在一个大小为 N 的域上提交了所有 N 个点的值。

[ANIM: 长条柱状图，N 个柱子]

第一轮折叠：验证者发送一个随机数 α。

[ANIM: 验证者发出 α]

证明者计算新多项式 f₁(x)，方法是把 f 的偶数项和奇数项用 α 线性组合。新多项式的度数变成了 d 除以 2。

[ANIM: 柱状图从 N 缩减为 N/2]

第二轮：验证者再发一个随机数 β。证明者再折一次，度数变成 d 除以 4。

[ANIM: 柱状图继续缩减到 N/4]

就这样反复折叠。每一轮，度数减半，域也缩小一半。

[ANIM: 快速折叠动画，柱子不断减半]

经过 log(N) 轮之后，多项式变成了一个常数。常数的验证是平凡的——直接比对就行。

[ANIM: 最后只剩一个柱子，绿色对勾]

最后，验证者还会随机抽查几个折叠步骤的一致性。如果都通过——证明有效！

[ANIM: 抽查箭头 + 全部通过]

### S10 WhyTransparentScene (7:20 - 8:00)

现在你可能会问：这和 SNARKs 有什么区别？

[ANIM: SNARKs vs STARKs 对比表]

最关键的区别是：SNARKs 需要一个「可信设置」。在初始化时，需要生成一些秘密参数，然后销毁。如果有人偷偷保留了这些参数，整个系统的安全性就崩溃了。

[ANIM: 可信设置 → 有毒废料问题]

而 STARKs 完全不需要可信设置。它只依赖哈希函数。所有参数都是公开透明的，没有任何秘密。

[ANIM: STARKs 侧全部透明绿色]

还有一个重要优势：STARKs 是抗量子的。因为它基于哈希函数，而不是椭圆曲线。即使未来量子计算机出现，STARKs 依然安全。

[ANIM: 量子计算机图标 + 盾牌]
